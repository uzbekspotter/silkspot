# Записка: защита Silkspot

Краткий чеклист, чтобы вернуться позже и пройти по пунктам. Не обязательно всё сразу — порядок от более критичного к «когда будет время».

---

## 1. Аккаунты и секреты (сделать в первую очередь)

- Включить **2FA** везде: GitHub, Vercel, Supabase, почта, регистратор домена, Cloudflare (если есть).
- Убедиться, что `**.env` и реальные ключи не в git** (проверить `.gitignore`, историю коммитов при необходимости).
- В Vercel / Supabase: **service role** и **R2 ключи** только в переменных окружения сервера, не в клиентском коде и не в `VITE_*`.

---

## 2. Supabase: авторизация и домен

- **Authentication → URL configuration**: Site URL и Redirect URLs под актуальный прод-домен (не только `*.vercel.app`).
- Убрать лишние wildcard-редиректы, если когда-то добавлялись «на широкую».
- Периодически смотреть **Auth logs** на подозрительную активность.

---

## 3. База: Row Level Security (RLS)

- В Dashboard: для **каждой таблицы** в `public` проверить, включён ли RLS. Особенно справочники: `countries`, `airports`, `airlines`, `aircraft_types`, `achievements` (каталог), `forum_categories`, `spotting_locations` — если RLS выключен и у `anon`/`authenticated` есть запись, это риск.
- Для справочников обычно: **RLS ON** + политика `SELECT` для всех (или только `authenticated`), **без** `INSERT`/`UPDATE`/`DELETE` для anon — правки только через service role или админ-RPC.

### Критично: профиль и роль

- Проверить в SQL Editor (тестовым пользователем), можно ли через клиент сделать  
`update user_profiles set role = 'ADMIN' where id = auth.uid()`.  
Если да — нужна **миграция**: запретить смену `role`, `is_banned` и защищённых счётчиков обычным пользователем (отдельный `WITH CHECK`, триггер `BEFORE UPDATE` или обновление только «безопасных» колонок через RPC).

## 4. API на Vercel (R2 presign / delete / upload)

Сейчас в коде **нет проверки**, что запрос идёт от залогиненного пользователя.

- `**/api/presign`**: требовать `Authorization: Bearer <access_token>`, валидировать сессию через Supabase; **жёстко проверять** `path` (префикс, формат, соответствие `user id` / правилам загрузки).
- `**/api/upload`** (fallback Vercel→R2, до ~4 MB): те же требования, что и к presign — сейчас открыт как presign; ограничение путей только `photos/YYYY/MM/…`.
- `**/api/delete`**: то же — только владелец или модератор, путь не произвольный.
- После смены домена: **CORS** — добавить свой origin в список разрешённых (или вынести в env), не только `silkspot.vercel.app`.

### R2: CORS на бакете (иначе браузерный PUT по presigned URL падает)

Загрузка идёт **напрямую из браузера в R2**. Если в консоли `ERR_CONNECTION_RESET` / «Upload connection failed», чаще всего не хватает правил CORS на **самом бакете** (не путать с CORS ответа `/api/presign` на Vercel).

1. Cloudflare Dashboard → R2 → ваш бакет (например `silkspot-photos`) → **Settings → CORS policy**.
2. Вставить правила по образцу `**docs/r2-bucket-cors.json`**, добавив в `AllowedOrigins` **точный** прод-URL (`https://ваш-домен`, без лишнего слэша).
3. Нужны методы `**PUT`** и заголовок `**Content-Type`** (как в примере). После сохранения подождать минуту и повторить загрузку.
4. **Каждый origin — отдельное значение**, без запятой *внутри* строки. Неверно: `http://localhost:3000,` как один origin (лишняя запятая ломает совпадение). Правильно: три отдельных origin: `http://localhost:3000`, `http://localhost:5173`, `https://silkspot.vercel.app`. Если открываешь сайт с **другого** URL (другой `*.vercel.app` или свой домен) — этот origin тоже добавь.
5. Если CORS уже как в пункте 3, а в консоли всё равно `**net::ERR_CONNECTION_RESET`** без явных CORS-ошибок, чаще виноваты **сеть / VPN / фаервол / провайдер** (обрыв до `*.r2.cloudflarestorage.com`), а не список origins. В коде есть **fallback `/api/upload`**: после неудачного прямого PUT клиент шлёт файл на Vercel, сервер кладёт объект в R2 (JPEG до ~4 MB).

Образец проверки роли/сессии уже есть в `api/r2-metrics.ts` — можно переиспользовать подход.

## 5. Зависимости и деплой

- Периодически: `npm audit`, обновления зависимостей по мере выхода патчей.
- Защитить **ветку main** в GitHub (review, статус CI при необходимости), чтобы прод не деплоился с произвольного коммита без контроля.

---

## 6. Cloudflare / инфраструктура (по желанию)

- Если домен за Cloudflare: **WAF**, rate limiting на чувствительные пути, минимизация утечки origin.
- R2: политика бакета, отдельные ключи с минимальными правами, ротация при компрометации.

## 7. Что уже в порядке (для спокойствия)

- На фронте только **anon**-ключ Supabase; RLS включён на основных пользовательских таблицах (фото, лайки, форум и т.д.) — см. `supabase/migrations/002_rls.sql`.
- `**/api/r2-metrics`** проверяет JWT и роль staff.
- `**/api/telegram-moderation`** рассчитан на секрет в заголовке — держать `TELEGRAM_WEBHOOK_SECRET` в env.
